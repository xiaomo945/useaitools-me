# 🛡️ 00号约束规则：神级专家创建铁律

## 🎯 适用范围
当用户要求创建或重做 `.trae/rules/` 目录下的神级专家文件（如 `01-programming.md`、`02-design.md` 等）时，本规则自动生效。

## ⚙️ 核心约束

### 1. 一次一个
**每次只创建或重做一个维度文件。** 无论用户指令中包含了多少个维度的名称或编号，只处理第一个或用户明确指定的那一个。完成后立即停止，等待用户的下一步指令。

### 2. 禁止批量
**绝不批量创建、预览或规划其他维度的文件。** 用户没有明确要求的维度，一个都不能动。即使在对话上下文中看到了其他维度的名称，也不得主动创建。

### 3. 等待确认
**每个维度文件创建完成后，必须等待用户确认。** 确认方式为告知用户当前进度（如"07号专家已创建，进度8/19"），并询问"继续创建下一个吗？"或等待用户主动给出下一个指令。不得在用户确认前自动开始下一个维度的创建。

### 4. 覆盖需确认
**如果需要覆盖已有的文件，必须提前告知用户。** 说明原因（如"当前版本质量不达标"），并等待用户明确同意后再执行覆盖操作。

### 5. 误判防护
**以下情况不属于任务结束，不得触发"下机"或"备份"流程**：
- 单个维度文件创建完成
- 用户短暂思考或切换任务
- 对话中出现"明天继续"等上下文讨论

**只有在用户明确说出"下机"、"收工"、"走了"、"备份素材"等自然语言触发词时，才执行对应的自动化流程。**

## 🚀 工作流规则：测试即推送

### 自动推送主分支
**每次完成任务并测试通过后，立即推送到主分支，避免积累未推送的更改。**

**执行流程**:
1. 完成任务
2. 运行构建测试: `npm run build`
3. 确认无错误
4. 提交更改: `git add . && git commit -m "feat/fix: [任务描述]"`
5. 推送到主分支: `git push origin HEAD:main`

**规则**:
- ✅ 测试通过 → 立即推送
- ❌ 测试失败 → 修复后再推送
- ⚠️ 不要积累多个任务后再推送

---

## 🧙 执行指令
当 `master-profile.md` 调用本维度时，以上述标准执行任务。本规则在所有神级专家创建任务中自动生效，优先级最高。
